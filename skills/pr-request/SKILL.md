# pr-request — 플랫폼팀 PR 리뷰 요청 메시지 생성

GitHub PR URL을 받아 **플랫폼 개발팀 톡방에 공유할 정중한 리뷰 요청 메시지**를 생성하고, 곧바로 macOS 클립보드에 복사한다. 사용자는 톡방에 붙여넣기(⌘V)만 하면 된다.

## 언제 쓰는가

- `github.com/.../pull/<N>` URL을 공유하며 "톡방에 올릴 PR 리뷰 요청 글 써줘" 라고 요청할 때
- `/pr-request <URL>` 명시적 호출
- "플랫폼팀에 리뷰 요청 글 만들어줘" 같은 자연어 요청

`/pr-review` 와 다른 점: `/pr-review` = 실제 코드 리뷰 생성. `/pr-request` = 톡방에 **올릴 요청 메시지**만 생성.

## 입력과 산출물

**입력**:
- GitHub PR URL (우선)
- URL 없으면 현재 git 브랜치의 열린 PR 자동 탐지 (`gh pr view --json ...`)

**산출물**:
1. macOS 클립보드에 복사된 리뷰 요청 메시지
2. 사용자에게 복사된 내용 **미리보기** 출력 (붙여넣기 전 확인용)

## 워크플로우

### 1단계 — PR 정보 수집

```bash
# URL 지정 시
gh pr view <URL> --json title,body,author,baseRefName,headRefName,url,additions,deletions,changedFiles,labels

# URL 미지정 시 (현재 브랜치)
gh pr view --json title,body,author,baseRefName,headRefName,url,additions,deletions,changedFiles,labels
```

인증 실패 시 `gh auth login` 안내하고 중단.

### 2단계 — 3줄 이내 요약 생성

PR title + body를 읽고 **최대 3줄** 로 요약.

**요약 작성 규칙**:
- 각 줄 50자 내외 (한국어 기준)
- 다음 중 의미 있는 것만 뽑는다 (3개 다 쓰려고 억지로 채우지 말 것 — 2줄로 충분하면 2줄)
  - **무엇이 바뀌는가**: 핵심 변경사항 1개
  - **왜 바뀌는가**: 배경/동기 (Jira 티켓, 버그, 요구사항)
  - **리뷰 포인트**: 특별히 봐줬으면 하는 부분 (위험 영역, 설계 판단)
- 인프라 PR이면 영향 레이어를 명시: `Terraform / K8s / ArgoCD / Helm values / 문서` 중 해당하는 것
- PR body가 영어여도 **한국어로** 요약
- 과장·자화자찬 금지 ("혁신적인" 등). 사실만 건조하게.

### 3단계 — 메시지 포맷팅

아래 템플릿을 기본으로 한다. 상황별로 약간 조정 가능.

**기본 템플릿**:

```
아래 PR 리뷰 부탁드립니다 🙇‍♂️

• <요약 1>
• <요약 2>
• <요약 3>

🔗 <PR URL>

감사합니다 🙏
```

**Jira 티켓이 있는 경우** — 첫 줄에 `[RDDP-xxx]` prefix:

```
[RDDP-544] PR 리뷰 부탁드립니다 🙇‍♂️

• <요약 1>
• <요약 2>

🔗 <PR URL>

감사합니다 🙏
```

**규모가 작은 PR** (문서/오탈자/설정값 변경):

```
간단한 PR이라 빠르게 리뷰 한 번만 봐주시면 감사하겠습니다 🙇

• <요약 1>

🔗 <PR URL>

감사합니다 🙏
```

**규모가 큰 PR** (`additions + deletions > 500` 이거나 `changedFiles > 20`) — 리뷰어 시간 양해:

```
분량이 조금 있는 PR이라 시간 되실 때 리뷰 부탁드립니다 🙇‍♂️

• <요약 1>
• <요약 2>
• <요약 3>

🔗 <PR URL>

감사합니다 🙏
```

### 4단계 — 클립보드 복사

macOS `pbcopy` 로 복사. `printf` 사용 (줄바꿈 유지 + trailing newline 제어):

```bash
printf '%s' "$MESSAGE" | pbcopy
```

복사 후 사용자에게 **반드시 프리뷰 출력**:

```
✅ 클립보드에 복사되었습니다. ⌘V 로 톡방에 붙여넣으세요.

─── 미리보기 ───
<메시지 전체>
────────────────
```

## 이모티콘 규칙

- **과하게 쓰지 않는다**. 👋 🙇‍♂️ 🔗 🙏 정도 조합으로 충분.
- 🚀 ✨ 💯 🔥 처럼 자기 업적 과시용 이모티콘 금지.
- 이모티콘은 톤을 **정중·담백**하게 유지하는 역할.

## 톤 규칙

- 한국어 존댓말 기본.
- 직접적이고 건조하게. "혹시 시간 괜찮으시면..." 같은 과도한 완곡어법 지양.
- 리뷰어에게 압박이 되지 않도록 "시간 되실 때" 류 여유 표현 1회 정도만 사용.

## 실패 케이스 대응

| 상황 | 대응 |
|---|---|
| `gh` 인증 실패 | `gh auth login` 안내 후 중단 |
| URL 없고 현재 브랜치에 PR 없음 | "현재 브랜치에 열린 PR이 없습니다. URL을 지정해주세요" 안내 |
| PR body가 너무 짧거나 비어있음 | title + diff 요약(`gh pr diff --name-only`)으로 대체 |
| PR이 draft 상태 | 메시지 맨 위에 `(draft PR)` 명시하고 "초안 단계라 가볍게 방향만 봐주시면 감사합니다" 톤으로 조정 |
| `pbcopy` 실패 (macOS 외 환경) | 메시지를 stdout에 출력하고 "수동 복사 필요" 안내 |

## 관련 스킬

- `/pr-review`: PR 자체의 코드 리뷰를 자동 생성. `/pr-request`는 그걸 "톡방에 올릴 요청 문구"로 변환하는 게 아니라 **별개의 요청 메시지 생성 단계** 이다.
- `/sdd-commit-pr`: 커밋/PR 본문 생성. `/pr-request`의 상위 공정이며 결과물이 달라짐 (PR 본문 vs 톡방 메시지).
